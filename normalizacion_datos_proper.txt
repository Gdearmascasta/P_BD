SHEET: Normalización (File: xl/worksheets/sheet2.xml)
====================================================
1) PRIMERA FORMA — Identificación de tablas y campos

- Tabla: ENCARGADO
- Campos: id_encargado, nombre, CC, edad

- Tabla: ZONA
- Campos: id_zona, codigo_zona, nombre_zona, temp_ambiente, humedad_ambiente

- Tabla: SENSOR
- Campos: id_sensor, codigo_sensor, nombre_sensor, modelo, id_zona

- Tabla: MEDIDA (también referida como MEDICION)
- Campos: id_medicion, fecha, hora, valor_temperatura, valor_humedad, id_sensor, id_zona, id_encargado

2) SEGUNDA FORMA — Repetición de tablas con PKs señaladas

- ENCARGADO: id_encargado [PK], nombre, CC, edad
- ZONA: id_zona [PK], codigo_zona, nombre_zona, temp_ambiente, humedad_ambiente
- SENSOR: id_sensor [PK], codigo_sensor, nombre_sensor, modelo, id_zona
- MEDIDA: id_medicion [PK], fecha, hora, valor_temperatura, valor_humedad, id_sensor, id_zona, id_encargado

- Observación sobre dependencias transitivas: No se detectan dependencias funcionales transitivas en el diseño actual; por tanto, NO existen DFT que requieran creación de tablas adicionales.

3) TERCERA FORMA — Tablas finales con PK y FK

- ENCARGADO
	- id_encargado [PK]
	- CC
	- nombre
	- edad

- ZONA
	- id_zona [PK]
	- codigo_zona
	- nombre_zona
	- temp_ambiente
	- humedad_ambiente

- SENSOR
	- id_sensor [PK]
	- codigo_sensor
	- nombre_sensor
	- modelo
	- id_zona [FK] -> ZONA(id_zona)

- MEDIDA
	- id_medicion [PK]
	- fecha
	- hora
	- valor_temperatura
	- valor_humedad
	- id_sensor [FK] -> SENSOR(id_sensor)
	- id_zona [FK] -> ZONA(id_zona)
	- id_encargado [FK] -> ENCARGADO(id_encargado)

- Nota sobre 4FN / relaciones M:N: No se identificaron relaciones Muchos-a-Muchos entre las entidades del modelo; por tanto, NO es necesario aplicar la 4ª forma ni crear tablas intermedias.

Resumen: 1FN lista tablas y campos; 2FN repite las tablas marcando las PK; 3FN muestra las tablas finales con sus PK y FK (sin tablas adicionales por transitividad).
